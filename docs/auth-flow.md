# 인증 및 사용자 흐름

## 인증 방식

DateLog v1에서는 Supabase Auth를 사용합니다. 로그인 방식은 이메일/비밀번호 회원가입과 이메일/비밀번호 로그인만 제공합니다.

v1에서는 다음 로그인 방식은 제공하지 않습니다.

- 카카오 로그인
- 구글 로그인
- 기타 OAuth 로그인

## 계정 생성 흐름

1. 사용자가 이메일과 비밀번호로 회원가입합니다.
2. Supabase Auth의 `auth.users`에 사용자 계정이 생성됩니다.
3. 서비스에서 사용하는 프로필 정보는 별도의 `profiles` 테이블에 저장합니다.
4. `profiles.id`는 `auth.users.id`와 동일한 uuid를 사용합니다.
5. 프로필 생성 후 사용자는 커플 생성/참여 상태를 확인합니다.

## 프로필 정책

`auth.users`는 인증 계정의 원본으로 사용합니다. 서비스 화면에서 필요한 사용자 이름, 표시 이름, 생성일 등의 정보는 `profiles` 테이블에 저장합니다.

`profiles.id`는 반드시 `auth.users.id`와 동일해야 합니다. 이를 통해 인증 사용자와 서비스 프로필을 1:1로 연결합니다.

## 로그인 후 라우팅

로그인한 사용자는 다음 상태에 따라 이동합니다.

- 커플이 있는 사용자: 데이트 기록 화면으로 이동합니다.
- 커플이 없는 사용자: 커플 생성/참여 화면으로 이동합니다.

커플이 없는 사용자는 데이트 기록 화면에 접근하지 못해야 합니다. 클라이언트 라우팅으로 막는 것뿐 아니라, 데이터 요청도 RLS 정책으로 보호해야 합니다.

## 비로그인 사용자 접근

비로그인 사용자는 랜딩 화면과 로그인/회원가입 화면만 접근할 수 있습니다.

비로그인 사용자가 데이트 기록, 커플 생성, 초대코드 참여 같은 인증이 필요한 화면에 접근하려고 하면 로그인 화면으로 이동해야 합니다.

## 커플 생성 흐름

1. 로그인한 사용자가 커플 생성 화면에 접근합니다.
2. 사용자가 커플을 생성합니다.
3. `couples` 테이블에 커플 row가 생성됩니다.
4. 생성자는 `couple_members`에 해당 `couple_id`의 멤버로 추가됩니다.
5. 생성자는 초대코드를 생성하거나 확인할 수 있습니다.
6. 생성된 초대코드는 상대방에게 전달됩니다.

## 초대코드 참여 흐름

1. 로그인한 사용자가 초대코드를 입력합니다.
2. 서버 또는 Supabase 정책으로 초대코드에 해당하는 커플을 확인합니다.
3. 해당 커플에 이미 2명이 참여했는지 확인합니다.
4. 사용자가 이미 다른 커플에 참여 중인지 확인합니다.
5. 조건을 만족하면 `couple_members`에 사용자를 추가합니다.
6. 참여 완료 후 데이트 기록 화면으로 이동합니다.

## 참여 제한

- 한 유저는 v1에서 하나의 커플만 참여할 수 있습니다.
- 한 커플은 최대 2명까지만 참여할 수 있습니다.
- 이미 2명이 참여한 커플에는 추가 참여할 수 없습니다.
- 이미 커플에 속한 사용자는 다른 초대코드로 추가 참여할 수 없습니다.

## 로그아웃 흐름

사용자가 로그아웃하면 Supabase Auth 세션을 종료합니다. 로그아웃 후에는 랜딩 또는 로그인 화면으로 이동해야 하며, 데이터 화면에는 접근할 수 없어야 합니다.
