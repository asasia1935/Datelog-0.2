# 보안 정책 및 RLS 설계

## 기본 원칙

DateLog v1에서는 RLS 정책을 필수로 적용합니다. 클라이언트 필터링만으로 권한을 처리하면 안 됩니다.

모든 커플 데이터는 `couple_id` 기준으로 분리합니다. 사용자는 자신이 속한 `couple_id`의 데이터만 조회, 작성, 수정, 삭제할 수 있어야 합니다.

## 인증 요구사항

- 로그인한 사용자만 커플 생성, 초대코드 참여, 데이트 기록 조회/작성/수정/삭제가 가능합니다.
- 비로그인 사용자는 랜딩 화면과 로그인/회원가입 화면 외의 데이터 화면에 접근할 수 없습니다.
- Supabase RLS 정책에서는 `auth.uid()`가 존재하는 사용자를 기준으로 접근을 허용해야 합니다.

## 멤버십 기반 접근 제어

RLS의 핵심 조건은 사용자가 `couple_members`에 속한 `couple_id`의 데이터만 접근할 수 있도록 하는 것입니다.

개념적으로 다음 조건을 반복 사용합니다.

```sql
exists (
  select 1
  from couple_members cm
  where cm.couple_id = target.couple_id
    and cm.user_id = auth.uid()
)
```

여기서 `target.couple_id`는 접근하려는 테이블의 `couple_id`를 의미합니다.

## profiles 정책

`profiles`는 사용자 본인의 프로필만 조회 및 수정할 수 있도록 제한합니다.

정책 방향:

- `select`: 로그인 사용자는 자신의 프로필을 조회할 수 있습니다.
- `insert`: 로그인 사용자는 자신의 `auth.uid()`와 같은 `id`로만 프로필을 생성할 수 있습니다.
- `update`: 로그인 사용자는 자신의 프로필만 수정할 수 있습니다.
- `delete`: v1에서는 프로필 삭제를 허용하지 않는 것이 권장됩니다.

핵심 조건:

```sql
id = auth.uid()
```

## couples 정책

`couples`는 해당 커플의 멤버만 조회할 수 있어야 합니다.

정책 방향:

- `select`: 사용자가 `couple_members`에 속한 커플만 조회할 수 있습니다.
- `insert`: 로그인 사용자는 커플을 생성할 수 있습니다.
- `update`: 해당 커플 멤버만 수정할 수 있습니다.
- `delete`: v1에서는 삭제를 제한하거나 owner만 허용하는 것이 권장됩니다.

커플 생성 시에는 생성자 본인이 `created_by = auth.uid()`가 되도록 강제해야 합니다.

## couple_members 정책

`couple_members`는 커플 멤버십의 권한 기준이므로 특히 엄격해야 합니다.

정책 방향:

- `select`: 사용자는 자신이 속한 커플의 멤버 목록만 조회할 수 있습니다.
- `insert`: 커플 생성 시 생성자 본인을 추가하거나, 유효한 초대코드 참여 흐름에서만 추가할 수 있습니다.
- `update`: v1에서는 일반 사용자의 멤버십 수정은 제한하는 것이 권장됩니다.
- `delete`: v1에서는 일반 사용자의 멤버십 삭제는 제한하는 것이 권장됩니다.

제약 조건:

- 한 사용자는 하나의 커플에만 속할 수 있어야 합니다.
- 한 커플은 최대 2명까지만 참여할 수 있어야 합니다.
- 이미 2명이 참여한 커플에는 추가 참여할 수 없어야 합니다.

이 제한은 RLS만으로 처리하기보다 unique 제약, 트랜잭션, RPC 함수 또는 DB trigger를 함께 사용해 보장하는 것이 권장됩니다.

## couple_invites 정책

초대코드는 커플 참여 권한과 직접 연결되므로 노출 범위를 제한해야 합니다.

정책 방향:

- `select`: 커플 멤버만 자신의 커플 초대코드를 조회할 수 있습니다.
- `insert`: 커플 멤버만 자신의 커플에 대한 초대코드를 생성할 수 있습니다.
- `update`: 초대코드 사용 처리 또는 만료 처리는 제한된 흐름에서만 허용합니다.
- `delete`: v1에서는 커플 멤버 또는 owner만 삭제할 수 있도록 제한하는 것이 권장됩니다.

초대코드 참여 시 확인해야 할 조건:

- 초대코드가 존재해야 합니다.
- 초대코드가 만료되지 않아야 합니다.
- 사용자가 아직 어떤 커플에도 참여하지 않았어야 합니다.
- 대상 커플의 멤버 수가 2명 미만이어야 합니다.

## date_logs 정책

데이트 기록은 `couple_id` 기준으로 접근을 제한합니다.

정책 방향:

- `select`: 사용자가 해당 `couple_id`의 멤버인 기록만 조회할 수 있습니다.
- `insert`: 사용자가 해당 `couple_id`의 멤버일 때만 작성할 수 있습니다.
- `update`: 사용자가 해당 `couple_id`의 멤버일 때만 수정할 수 있습니다.
- `delete`: 사용자가 해당 `couple_id`의 멤버일 때만 삭제할 수 있습니다.

핵심 조건:

```sql
exists (
  select 1
  from couple_members cm
  where cm.couple_id = date_logs.couple_id
    and cm.user_id = auth.uid()
)
```

`insert` 시에는 `created_by = auth.uid()` 조건도 함께 강제하는 것이 권장됩니다.

## log_photos 정책

사진 메타데이터는 `date_logs`와 동일하게 `couple_id` 기준으로 접근을 제한합니다.

정책 방향:

- `select`: 사용자가 해당 `couple_id`의 멤버인 사진만 조회할 수 있습니다.
- `insert`: 사용자가 해당 `couple_id`의 멤버이고, 연결된 `log_id`도 같은 `couple_id`에 속할 때만 허용합니다.
- `update`: 사용자가 해당 `couple_id`의 멤버일 때만 허용합니다.
- `delete`: 사용자가 해당 `couple_id`의 멤버일 때만 허용합니다.

기록당 최대 3장 제한은 DB 제약, trigger, 또는 제한된 RPC 함수를 통해 보장하는 것이 권장됩니다.

## Storage 정책

사진 파일은 Supabase Storage에 저장합니다. 경로는 다음 형식을 사용합니다.

```text
couples/{coupleId}/logs/{logId}/{fileName}
```

Storage 접근 정책은 path의 `coupleId`를 기준으로 사용자가 해당 커플의 멤버인지 확인해야 합니다.

정책 방향:

- `select`: 해당 `coupleId`의 멤버만 파일을 읽을 수 있습니다.
- `insert`: 해당 `coupleId`의 멤버만 파일을 업로드할 수 있습니다.
- `update`: 해당 `coupleId`의 멤버만 파일을 교체할 수 있습니다.
- `delete`: 해당 `coupleId`의 멤버만 파일을 삭제할 수 있습니다.

Storage 파일과 `log_photos.storage_path`가 불일치하지 않도록, 업로드 완료 후 메타데이터 저장 순서를 명확히 해야 합니다.

## 클라이언트 보안 기준

클라이언트에서는 사용자 경험을 위해 화면 접근을 제한할 수 있습니다. 그러나 보안의 최종 책임은 RLS와 DB 제약에 있어야 합니다.

다음 상황은 반드시 서버/DB 정책에서 차단되어야 합니다.

- 사용자가 다른 커플의 `couple_id`로 기록 조회를 시도하는 경우
- 사용자가 다른 커플의 `couple_id`로 기록 작성을 시도하는 경우
- 사용자가 다른 커플의 기록 ID로 수정 또는 삭제를 시도하는 경우
- 이미 커플에 속한 사용자가 다른 커플 초대코드로 참여하려는 경우
- 이미 2명이 참여한 커플에 추가 사용자가 참여하려는 경우
- 기록당 3장을 초과해 사진을 등록하려는 경우
