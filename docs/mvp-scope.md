# DateLog MVP 범위

## 목적

DateLog는 커플 전용 데이트 기록 서비스입니다. v1의 목표는 5커플, 총 10명 정도가 실제로 사용할 수 있는 작은 규모의 MVP를 안정적으로 제공하는 것입니다.

v1에서는 기능 범위를 좁게 유지하고, 커플 단위 데이터 분리와 기본적인 기록 작성 경험을 우선합니다. Supabase 연동 전 이 문서는 구현 범위를 고정하기 위한 기준으로 사용합니다.

## v1 포함 범위

- Supabase Auth 기반 이메일/비밀번호 회원가입
- Supabase Auth 기반 이메일/비밀번호 로그인
- 서비스 프로필 정보 저장
- 커플 생성
- 초대코드 생성
- 초대코드를 통한 커플 참여
- 커플별 데이트 기록 조회
- 커플별 데이트 기록 작성
- 커플별 데이트 기록 수정
- 커플별 데이트 기록 삭제
- 기록당 최대 3장의 사진 업로드
- Supabase Storage를 사용한 사진 저장
- Row Level Security 기반 데이터 접근 제어

## v1 제외 범위

- 카카오 로그인
- 구글 로그인
- 기타 소셜 로그인
- 공개 피드
- 여러 커플 참여
- 커플 3명 이상 참여
- 비로그인 상태의 데이터 화면 접근
- 관리자 대시보드
- 결제
- 알림
- 공유 링크 기반 공개 열람
- 고급 검색 및 통계

## 사용자 및 커플 제약

- 한 유저는 v1에서 하나의 커플만 참여할 수 있습니다.
- 한 커플은 최대 2명까지만 참여할 수 있습니다.
- 커플 생성자는 초대코드를 만들 수 있습니다.
- 상대방은 초대코드로 해당 커플에 참여합니다.
- 이미 2명이 참여한 커플에는 추가 사용자가 참여할 수 없습니다.
- 커플이 없는 사용자는 데이트 기록 화면에 접근할 수 없으며, 커플 생성/참여 화면으로 이동해야 합니다.

## 접근 제어 원칙

- 로그인한 사용자만 커플 생성, 초대코드 참여, 데이트 기록 조회/작성/수정/삭제가 가능합니다.
- 비로그인 사용자는 랜딩 화면과 로그인/회원가입 화면 외의 데이터 화면에 접근할 수 없습니다.
- 데이트 기록은 `couple_id` 기준으로 분리합니다.
- 사용자는 자신이 속한 `couple_id`의 기록만 조회/작성/수정/삭제할 수 있어야 합니다.
- 권한 처리는 클라이언트 필터링에 의존하지 않습니다.
- Supabase RLS 정책을 필수로 적용합니다.

## 사진 저장 정책

- 사진은 Supabase Storage에 저장합니다.
- 초기 MVP에서는 기록당 최대 3장으로 제한합니다.
- Storage path는 다음 형식을 사용합니다.

```text
couples/{coupleId}/logs/{logId}/{fileName}
```

## v1 성공 기준

- 5커플/10명 규모에서 회원가입, 로그인, 커플 생성/참여, 기록 작성 흐름이 안정적으로 동작해야 합니다.
- 사용자가 자신의 커플 데이터 외의 기록을 조회하거나 수정할 수 없어야 합니다.
- 클라이언트에서 잘못된 `couple_id`를 전달하더라도 RLS 정책으로 차단되어야 합니다.
- 사진 업로드는 기록당 3장 제한과 커플별 경로 분리를 지켜야 합니다.
